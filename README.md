# devsec-demo 🔒

A hands-on demonstration of **Secure Software Development Lifecycle (SSDLC)**
practices, built as a reference for the [Pelino Courses](https://github.com/Pelino-Courses)
DevSec curriculum.

The repository contains a small but realistic Python/Flask web application
(**SecureNotes**) wired up with the tooling and processes that make up a
modern SSDLC.

## What is an SSDLC?

The **Secure Software Development Lifecycle** integrates security at every
phase of development – from design through deployment – rather than treating
it as an afterthought.

```
Plan → Design → Develop → Test → Release → Operate
  ↑                                            |
  └──────────── Feedback & Monitoring ─────────┘
```

Key phases and how they map to this repo:

| SSDLC Phase | Practice | Where in this repo |
|-------------|----------|--------------------|
| **Plan** | Threat modelling, security requirements | This README, SECURITY.md |
| **Develop** | Secure coding guidelines, peer review | `app/`, CODEOWNERS |
| **Test** | SAST, unit tests for security controls | `tests/`, CI workflow |
| **Release** | Dependency audit, secrets scanning | CI workflow, Dependabot |
| **Operate** | Vulnerability disclosure process | SECURITY.md |

## Security practices in this repo

### 1 · Password hashing (bcrypt)
Passwords are **never stored in plain text**.  The `User` model uses
[Flask-Bcrypt](https://flask-bcrypt.readthedocs.io/) with a work-factor of 12
so that brute-force attacks remain computationally expensive.

```python
# app/models.py
def set_password(self, password: str) -> None:
    self.password_hash = bcrypt.generate_password_hash(password).decode("utf-8")
```

### 2 · CSRF protection
Every HTML form includes a hidden CSRF token generated by
[Flask-WTF](https://flask-wtf.readthedocs.io/).  POST requests without a
valid token are rejected with HTTP 400.

### 3 · Parameterised queries (SQL injection prevention)
All database access goes through the SQLAlchemy ORM.  Raw SQL strings are
never used, so user-supplied input cannot alter query structure.

### 4 · IDOR prevention
Every note operation checks `note.user_id == current_user.id` before
proceeding.  Accessing another user's resource returns HTTP 403.

```python
# app/routes.py
if note.user_id != current_user.id:
    abort(403)
```

### 5 · Secure HTTP headers
[Flask-Talisman](https://github.com/GoogleCloudPlatform/flask-talisman) adds:
- `Content-Security-Policy` (restrict script/style origins)
- `Strict-Transport-Security` (HTTPS enforcement in production)
- `X-Frame-Options: DENY` (clickjacking protection)
- `X-Content-Type-Options: nosniff`

### 6 · Rate limiting
[Flask-Limiter](https://flask-limiter.readthedocs.io/) restricts the login
endpoint to **10 requests per minute per IP**, defending against brute-force
and credential-stuffing attacks.

### 7 · Open-redirect prevention
After login the `next` redirect parameter is validated with `urlparse`; any
URL with a non-empty `netloc` (i.e. an absolute URL) is discarded.

### 8 · Secrets management
No secrets are hard-coded.  The application reads `SECRET_KEY` and
`DATABASE_URL` from environment variables and refuses to start if
`SECRET_KEY` is absent.  A `.env.example` template is committed; the real
`.env` is git-ignored.

### 9 · Secure session cookies
| Cookie attribute | Value | Purpose |
|------------------|-------|---------|
| `HttpOnly` | `True` | Prevent JavaScript access |
| `Secure` | `True` (production) | HTTPS only |
| `SameSite` | `Lax` | CSRF mitigation |

### 10 · Static Application Security Testing (SAST)
The CI pipeline runs [Bandit](https://bandit.readthedocs.io/) on every push
to identify common Python security issues (e.g., use of `eval`, hardcoded
passwords, weak hashing algorithms).

### 11 · CodeQL analysis
GitHub's [CodeQL](https://codeql.github.com/) performs deep semantic analysis
of the codebase on every PR to `main` and on a weekly schedule, reporting
results in the Security tab.

### 12 · Dependency auditing (pip-audit + Dependabot)
- **pip-audit** runs in CI on every push and checks all runtime dependencies
  against the [OSV](https://osv.dev/) vulnerability database.
- **Dependabot** opens automated PRs to update outdated or vulnerable
  dependencies every Monday.

### 13 · Mandatory code review (CODEOWNERS)
Security-sensitive files (authentication, configuration, CI workflows) require
review from the `@Pelino-Courses/security` team before a PR can be merged.

### 14 · Vulnerability disclosure policy
`SECURITY.md` defines a responsible-disclosure process so that external
researchers know how to report issues privately.

## Getting started

### Prerequisites

- Python 3.11+
- (Optional) [virtualenv](https://virtualenv.pypa.io/) or the built-in `venv`

### Setup

```bash
# 1. Clone the repository
git clone https://github.com/Pelino-Courses/devsec-demo.git
cd devsec-demo

# 2. Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements-dev.txt

# 4. Configure environment variables
cp .env.example .env
# Edit .env and set a strong SECRET_KEY:
#   python -c "import secrets; print(secrets.token_hex(32))"

# 5. Run the development server
FLASK_ENV=development python run.py
```

Open http://127.0.0.1:5000 in your browser.

## Running the tests

```bash
# Run all tests with coverage report
pytest --cov=app --cov-report=term-missing

# Run only security-focused tests
pytest tests/test_auth.py tests/test_routes.py -v

# Static security analysis
bandit -r app/ -ll

# Dependency vulnerability audit
pip-audit -r requirements.txt
```

## Contributing

1. Fork the repository and create a feature branch.
2. Follow the secure coding guidelines described above.
3. Ensure all tests pass and no new bandit warnings are introduced.
4. Open a Pull Request – CODEOWNERS will route it to the right reviewers.

To report a security vulnerability, see [SECURITY.md](SECURITY.md).
